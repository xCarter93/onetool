## Summary

This phase replaces the deterministic synonym/substring matching inside `mapSchemaTool.execute()` with an LLM call that returns structured output. The existing tool interface (input: headers, sampleRows, entityType; output: mappings, unmappedColumns, missingRequiredFields) stays identical -- only the internal mapping logic changes.

The cleanest approach is to call the AI SDK `generateObject()` directly inside the tool's `execute()` function, bypassing the Mastra agent entirely. The project already has `ai` v6.0.27 and `@ai-sdk/openai` v3.0.7 installed. The `generateObject` function accepts a Zod schema and returns validated structured data in a single call -- exactly what we need. This avoids the overhead of instantiating a Mastra Agent just for one structured extraction.

**Primary recommendation:** Use `generateObject` from the `ai` SDK directly inside `mapSchemaTool.execute()` with `openai("gpt-5-nano")` and a Zod response schema. Delete all deterministic matching code.

## Architecture Patterns

### Recommended: Direct `generateObject` inside tool execute

The user's CONTEXT.md says "Use Mastra's structured output via generate() with a Zod schema." However, user also gave Claude discretion on "whether to use agent.generate() or call LLM directly inside tool execute()."

**Recommendation: Use `generateObject` from the `ai` SDK directly.** Rationale:
1. The API route already calls `mapSchemaTool.execute()` directly -- no agent involved
2. Creating a Mastra Agent just to call `agent.generate()` once is unnecessary indirection
3. `generateObject` is the standard AI SDK pattern for "give me structured data from a prompt"
4. The existing project already has `ai` v6 and `@ai-sdk/openai` installed
5. The Mastra agent's `generate()` internally delegates to the AI SDK anyway

### Pattern: generateObject inside tool execute

**What:** Call `generateObject` with a Zod schema from inside the tool's execute function
**When to use:** Whenever a tool needs LLM-powered structured extraction

```typescript
// Source: AI SDK docs (https://ai-sdk.dev/docs/ai-sdk-core/generating-structured-data)
import { generateObject } from "ai";
import { openai } from "@ai-sdk/openai";
import { z } from "zod";

const llmResponseSchema = z.object({
  mappings: z.array(
    z.object({
      csvColumn: z.string(),
      schemaField: z.string(),
      confidence: z.number().min(0).max(1),
    })
  ),
  unmappedColumns: z.array(z.string()),
});

const { object } = await generateObject({
  model: openai("gpt-5-nano"),
  schema: llmResponseSchema,
  prompt: "...",
  abortSignal: AbortSignal.timeout(15_000),
});
```

### Pattern: Timeout with AbortSignal

**What:** Use `AbortSignal.timeout()` to enforce the 15-second timeout
**When to use:** For the LLM call inside tool execute

```typescript
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), 15_000);

try {
  const { object } = await generateObject({
    model: openai("gpt-5-nano"),
    schema: llmResponseSchema,
    prompt: buildMappingPrompt(headers, sampleRows, schemaFields),
    abortSignal: controller.signal,
  });
  clearTimeout(timeoutId);
  return object;
} catch (error) {
  clearTimeout(timeoutId);
  // Return graceful fallback (all unmapped)
  throw error; // or handle per user's error strategy
}
```

**Note:** `AbortSignal.timeout(15_000)` is simpler if supported in the runtime. Both patterns work.

### Pattern: Error handling -- graceful degradation to manual mapping

**What:** On any LLM failure, return all columns as unmapped
**When to use:** LLM timeout, API error, Zod validation failure

```typescript
try {

    mappings: [],
    unmappedColumns: headers,
```

### Anti-Patterns to Avoid
- **Creating a dedicated Agent just for structured output:** Overkill for a single `generateObject` call. Agents are for multi-turn conversation and tool orchestration.
- **Keeping any deterministic fallback code:** User explicitly chose no fallback. Delete it all.
- **Wrapping the LLM call in another abstraction layer:** Keep it simple -- `generateObject` directly in execute.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Structured LLM output | Manual JSON.parse + validation | `generateObject` with Zod schema | Handles retries, validation, type safety automatically |
| Timeout handling | Manual Promise.race | `AbortSignal.timeout(15_000)` or `abortSignal` param | Standard API, cleaner error propagation |
| Schema formatting for prompt | Complex template engine | JSON.stringify of CLIENT_SCHEMA_FIELDS | The schema is already a well-structured const object |

**Key insight:** The AI SDK's `generateObject` handles the entire structured output pipeline -- schema injection into the API call, response parsing, Zod validation. Don't replicate any of that manually.

## Common Pitfalls

### Pitfall 1: OpenAI Structured Output -- optional fields not supported
**What goes wrong:** OpenAI's structured output mode (used by `generateObject`) does not support `z.optional()`. It requires `z.nullable()` instead.
**Why it happens:** OpenAI's JSON schema implementation has stricter requirements than standard JSON Schema.
**How to avoid:** Use `.nullable()` instead of `.optional()` for any optional fields in the LLM response schema. The `sampleValue` field in the mapping must be `z.string().nullable()` not `z.string().optional()`.
**Warning signs:** Zod validation errors or OpenAI API errors about unsupported schema features.

### Pitfall 2: Schema field names with dots confuse LLM
**What goes wrong:** Fields like `contact.firstName` and `property.streetAddress` use dots as namespace separators, which an LLM might misinterpret.
**Why it happens:** Dots in field names are unusual -- LLMs may treat them as object paths.
**How to avoid:** In the prompt, explicitly explain the dot-namespace convention and list all valid field names. Include examples like "contact.firstName means the contact's first name field, not a nested object."
**Warning signs:** LLM returns `firstName` instead of `contact.firstName`.

### Pitfall 3: LLM invents schema fields that don't exist
**What goes wrong:** LLM maps a CSV column to a schema field name that doesn't exist in CLIENT_SCHEMA_FIELDS.
**Why it happens:** LLM hallucination -- it guesses plausible field names.
**How to avoid:** Post-process the LLM response: validate each `schemaField` against the actual schema keys. Move any mapping with an invalid schemaField to `unmappedColumns`. Also include in the prompt: "You MUST only use field names from the provided schema. Do not invent field names."
**Warning signs:** Downstream validation errors when trying to use nonexistent field names.

### Pitfall 4: Duplicate schema field assignments
**What goes wrong:** LLM maps two different CSV columns to the same schema field.
**Why it happens:** Similar column names (e.g., "Name" and "Company Name" both mapped to `companyName`).
**How to avoid:** Post-process: if a schemaField appears more than once, keep the highest-confidence mapping and move the other to `unmappedColumns`. Include in prompt: "Each schema field should be mapped to at most one CSV column."
**Warning signs:** Two mappings with the same schemaField in the output.

### Pitfall 5: Token budget -- large schema + many headers
**What goes wrong:** With 24 schema fields + 20+ CSV headers + sample rows, the prompt could get large.
**Why it happens:** GPT-5 nano is the smallest model -- context window may be limited.
**How to avoid:** GPT-5 nano supports 128K context, so this is unlikely to be an issue for CSV mapping. The full prompt should be well under 2K tokens. Still, keep the prompt concise.
**Warning signs:** Truncated responses or API errors about context length.

### Pitfall 6: Mastra tool execute return type union
**What goes wrong:** `mapSchemaTool.execute()` returns a union type that includes `ValidationError`. The API route already handles this with the `"error" in mapRaw` check.
**Why it happens:** Mastra's `createTool` wraps execute results in a union.
**How to avoid:** The error handling inside the tool's execute should throw or return the expected output shape. The API route's existing error check will catch validation errors from Mastra's wrapper. No change needed in the route.
**Warning signs:** Type errors when accessing properties on the return value.

## Code Examples

### Complete mapSchemaTool rewrite pattern

```typescript
// Source: AI SDK docs + project patterns
import { createTool } from "@mastra/core/tools";
import { generateObject } from "ai";
import { openai } from "@ai-sdk/openai";
import { z } from "zod";
import { CLIENT_SCHEMA_FIELDS, PROJECT_SCHEMA_FIELDS } from "@/types/csv-import";

// Zod schema for LLM structured output
// NOTE: Use .nullable() not .optional() for OpenAI compatibility
const llmMappingSchema = z.object({
  mappings: z.array(
    z.object({
      csvColumn: z.string(),
      schemaField: z.string(),
      confidence: z.number().min(0).max(1),
      sampleValue: z.string().nullable(),
    })
  ),
  unmappedColumns: z.array(z.string()),
});

export const mapSchemaTool = createTool({
  id: "map-schema",
  description: "Map CSV column headers to Convex schema fields...",
  inputSchema: z.object({
    entityType: z.enum(["clients", "projects"]),
    headers: z.array(z.string()),
    sampleRows: z.array(z.record(z.string(), z.string())).optional(),
  }),
  outputSchema: z.object({
    mappings: z.array(
      z.object({
        csvColumn: z.string(),
        schemaField: z.string(),
        confidence: z.number().min(0).max(1),
        dataType: z.string(),
        isRequired: z.boolean(),
        sampleValue: z.string().optional(),
      })
    ),
    unmappedColumns: z.array(z.string()),
    missingRequiredFields: z.array(z.string()),
  }),
  execute: async (input) => {
    const { entityType, headers, sampleRows } = input;
    const schema = entityType === "clients" ? CLIENT_SCHEMA_FIELDS : PROJECT_SCHEMA_FIELDS;

    try {
      const { object } = await generateObject({
        model: openai("gpt-5-nano"),
        schema: llmMappingSchema,
        prompt: buildMappingPrompt(entityType, headers, sampleRows, schema),
        abortSignal: AbortSignal.timeout(15_000),
      });

      // Post-process: validate schemaField names, enrich with dataType/isRequired
      return postProcessMappings(object, headers, schema);
    } catch (error) {
      // LLM failure -- return all columns as unmapped (user maps manually)
      return {
        mappings: [],
        unmappedColumns: [...headers],
        missingRequiredFields: Object.entries(schema)
          .filter(([, info]) => info.required)
          .map(([name]) => name),
      };
    }
  },
});
```

### Prompt construction pattern

```typescript
function buildMappingPrompt(
  entityType: string,
  headers: string[],
  sampleRows: Record<string, string>[] | undefined,
  schema: Record<string, { type: string; required: boolean; group?: string; options?: readonly string[] }>
): string {
  const schemaDescription = Object.entries(schema).map(([name, info]) => ({
    fieldName: name,
    type: info.type,
    required: info.required,
    ...(info.options ? { allowedValues: info.options } : {}),
  }));

  return `You are mapping CSV columns to database schema fields for a ${entityType} import.

## Schema Fields (use ONLY these exact field names)
${JSON.stringify(schemaDescription, null, 2)}

## CSV Headers
${JSON.stringify(headers)}

## Sample Data (first ${sampleRows?.length ?? 0} rows)
${sampleRows ? JSON.stringify(sampleRows.slice(0, 5), null, 2) : "No sample data provided"}

## Instructions
- Map each CSV header to the most appropriate schema field
- Fields like "contact.firstName" use dot-notation as a namespace (not nested objects) -- use the exact string including the dot
- Each schema field can be mapped to at most one CSV column
- Provide a confidence score (0.0-1.0) for each mapping based on how well the header/data matches
- If a CSV column doesn't match any schema field, include it in unmappedColumns
- Use sample data to disambiguate when headers are ambiguous
- For sampleValue, use the value from the first sample row for that column (or null if unavailable)`;
}
```

### Post-processing pattern

```typescript
function postProcessMappings(
  llmResult: z.infer<typeof llmMappingSchema>,
  headers: string[],
  schema: Record<string, { type: string; required: boolean }>
) {
  const validFieldNames = new Set(Object.keys(schema));
  const usedFields = new Set<string>();
  const mappings: Array<{
    csvColumn: string;
    schemaField: string;
    confidence: number;
    dataType: string;
    isRequired: boolean;
    sampleValue?: string;
  }> = [];
  const unmappedColumns = new Set(llmResult.unmappedColumns);

  // Sort by confidence descending so higher-confidence mappings win duplicates
  const sorted = [...llmResult.mappings].sort((a, b) => b.confidence - a.confidence);

  for (const mapping of sorted) {
    // Validate: schemaField must exist and not already be used
    if (!validFieldNames.has(mapping.schemaField) || usedFields.has(mapping.schemaField)) {
      unmappedColumns.add(mapping.csvColumn);
      continue;
    }

    // Skip low-confidence below threshold (keep in mappings but they'll show as "suggested")
    usedFields.add(mapping.schemaField);
    const fieldInfo = schema[mapping.schemaField];
    mappings.push({
      csvColumn: mapping.csvColumn,
      schemaField: mapping.schemaField,
      confidence: mapping.confidence,
      dataType: String(fieldInfo.type),
      isRequired: Boolean(fieldInfo.required),
      sampleValue: mapping.sampleValue ?? undefined,
    });
  }

  // Any header not in mappings and not already in unmapped
  const mappedCsvColumns = new Set(mappings.map((m) => m.csvColumn));
  for (const header of headers) {
    if (!mappedCsvColumns.has(header)) {
      unmappedColumns.add(header);
    }
  }

  // Missing required fields
  const missingRequiredFields = Object.entries(schema)
    .filter(([name, info]) => info.required && !usedFields.has(name))
    .map(([name]) => name);

  return {
    mappings,
    unmappedColumns: [...unmappedColumns],
    missingRequiredFields,
  };
}
```

## Open Questions

1. **GPT-5 nano model string**
   - What we know: OpenAI docs list `gpt-5-nano` as the model identifier
   - What's unclear: Whether `@ai-sdk/openai` v3.0.7 supports this model string (it should -- it passes model strings through to the API)
   - Recommendation: Use `openai("gpt-5-nano")`. If it fails at runtime, fall back to `openai("gpt-4o-mini")` as a known-working alternative.

2. **Error surface to user**
   - What we know: On LLM failure, return empty mappings + all columns unmapped
   - What's unclear: Should the tool throw (letting the API route catch it and return 500) or return the empty-mapping result (200 with no auto-mappings)?
   - Recommendation: Return the empty-mapping result as a success (200). The UI already handles unmapped columns. The user decision says "show error and let user map manually" -- returning unmapped columns IS the manual mapping state. Log the error server-side.
